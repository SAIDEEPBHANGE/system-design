# 🧠 Project Notes - Global Distributed URL Shortener (Highly Available & Fault-Tolerant)

---

## 📌 What I understood from this project

- This system is a globally distributed, high-throughput URL shortener designed to handle billions of links and trillions of click events with minimal latency.
- It provides instant redirection for users worldwide by deploying services across multiple regions (Region A & Region B) with a zero Single Point of Failure (SPOF) architecture.
- It intelligently separates the core heavy transactional traffic (Reads/Writes) from the analytical tracking traffic using an asynchronous event-driven pipeline.

---

## 🤔 My Initial Thinking

Before designing:

- I initially thought the system would be a simple CRUD service with a standard relational database (SQL) storing `long_url` and `short_url` mappings.
- I assumed a traditional background cron job would be enough to scan the database daily and delete expired links.
- I thought analytics could be counted by just doing an `UPDATE clicks = clicks + 1` query directly inside the primary database whenever a link was clicked.

---

## 🏗️ What I actually built (V01)

- **Simple CRUD system:** Basic endpoints to generate a short key and redirect users.
- **Single backend server:** A monolithic application handling both URL shortening logic and redirection.
- **Single database:** A localized database setup without sharding, global replication, or multi-region failover.
- **No caching / scaling:** No Redis layer to handle heavy read spikes on viral links, making the database vulnerable to connection exhaustion.

---

## 🔄 Evolution of Design

### V01

- Basic monolithic system with a single database instance.
- No scaling considerations, highly vulnerable to single-point failures.

### V02

- Separated the system into dedicated **Read Services** (for high-speed redirection) and **Write Services** (for link generation).
- Introduced **Distributed NoSQL Sharding** based on `hash(short_code)` to distribute the database load horizontally.
- Implemented **Time-To-Live (TTL)** directly at the database layer to automatically purge expired ephemeral links instead of running heavy batch cron jobs.

### V03 (The "God-Level" Architecture)

- **Multi-Region Active-Active Deployment:** Deployed autonomous application instances across multiple global regions backed by a Global Traffic Manager (GTM) for geo-routing and automated failover.
- **Distributed Key Generation Service (KGS):** Built a standalone KGS with a _Key Range Buffer_ that pre-allocates unique token ranges to each region, completely eliminating cross-region coordination overhead and ID collisions.
- **Asynchronous Event-Driven Analytics:** Integrated **Apache Kafka** to stream click logs silently into an OLAP Data Warehouse, ensuring that heavy analytical queries never slow down the core redirection workflow.

---

## ⚙️ Key Concepts I Learned

- **Multi-Region High Availability (HA):** How to route traffic dynamically using Global Load Balancers and manage active-active service failovers.
- **Distributed ID Generation:** Using a KGS with token buffering to achieve lock-free unique key generation at scale.
- **Database Sharding & TTL:** Partitioning NoSQL data to break hardware limitations and leveraging native database TTL for predictable resource consumption.
- **Asynchronous Pipelines (Kafka):** Decoupling critical user-facing flows (Shortening/Redirection) from non-critical resource-heavy flows (Analytics logging).
- **System Observability:** Setting up dedicated Time-Series schemas/databases to monitor system metrics (latency, error rates, KGS buffers) separate from business logic.

---

## ⚠️ Mistakes I made

- **Assuming Single-DB Capacity:** Underestimating the volume of write queries generated by global short-link creation and the read spikes on viral links.
- **Batch Deletion Overhead:** Thinking a standard background cron job could clean up millions of expired rows without causing severe database CPU spikes and locking issues.
- **Mixing Logs with Core Data:** Initially planning to write system metrics and user click logs into the primary OLTP database, which would have rapidly degraded performance.

---

## 🧩 Questions That Confused Me

- **Active-Active Cross-Region Replication Latency:** _How do we sync databases across the globe without facing race conditions?_ (Solved by enforcing unique, pre-allocated key ranges via KGS per region).
- **Batch Processing vs Database TTL:** _Why not use traditional scripts to clean up guest URLs?_ (Learned that database-level native TTL spreads the deletion load evenly and continuously, avoiding execution spikes).
- **OLTP vs OLAP:** _Why can't we just query click history from the main database?_ (Understood that transactional databases are for quick row lookups, whereas analytics need specialized columnar storage for aggregation queries).

---

## 💡 Improvements I discovered later

- Always split user profiles and data into distinct categories, such as **Permanent (User-linked)** vs **Ephemeral (Guest)** links, to optimize storage strategies early.
- High availability is not just about backing up code; it requires robust circuit breakers, rate limiting at the API Gateway, and strict health checks at the Load Balancer level.
- Observability (Metrics, Logs, Traces) is a core component of system architecture, not an afterthought. A system is only as good as its monitoring stack.

---

## 🚀 What I would do differently next time

- **Start with Non-Functional Requirements (NFRs):** Quantify the exact scale (DAU, QPS, Storage capacity for 5 years) before sketching any system blocks.
- **Design Core Interfaces First:** Draft the REST/gRPC API contracts and JSON payloads prior to laying out database structures.
- **Failure Mode and Effects Analysis (FMEA):** Map out "What if Region A goes down?", "What if Kafka lags?", or "What if Redis cache evicts keys prematurely?" right at the beginning.

---

## 📌 Final Understanding (after completion)

- **System design is a game of trade-offs:** You compromise absolute real-time data consistency across regions to achieve extreme global availability and ultra-low latency.
- **Every component must justify its cost:** A cache solves read latency, a message queue solves write buffering/decoupling, and sharding solves storage bottlenecks—each must have a distinct purpose.
- **Architectures are living organisms:** They naturally evolve from simple V01 CRUD code into highly resilient, distributed V03 meshes as the requirements scale up.
